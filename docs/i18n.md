# Internationalization (i18n)

The web app uses a lightweight message-catalog pattern — no `next-intl` or heavy framework. UI strings live in typed nested objects; components call `t("namespace.key", { params })`.

Engine-generated text (match explanations, exclusions, red-flag labels) uses **JSON locale packs** in `@aperio-j/core` — see [engine-locales.md](./engine-locales.md).

## Locales

| Code | UI catalog | Engine JSON | Default |
|------|------------|-------------|---------|
| `en` | `apps/web/src/i18n/messages/en.ts` | `packages/core/locales/en.json` | **Yes** |
| `zh-CN` | `apps/web/src/i18n/messages/zh-CN.ts` | `packages/core/locales/zh-CN.json` | |
| `es` | `apps/web/src/i18n/messages/es.ts` | `packages/core/locales/es.json` | Partial UI overlay |

Locale resolution order:

1. Cookie `aperio_j_locale` (set by header locale switcher)
2. `Accept-Language` request header (SSR first visit)
3. `en`

The same resolved locale is passed to the match pipeline so inbox explanations follow the UI language.

## Layout

```
apps/web/src/i18n/
  messages/zh-CN.ts   # Messages type source of truth
  messages/en.ts      # English catalog (default)
  messages/es.ts      # Español overlay (spreads en)
  catalog.ts          # getMessages(locale)
  translate.ts        # createTranslator, interpolate, resolveLocale
  server.ts           # getServerLocale(), getServerTranslator()
  provider.tsx        # I18nProvider, useI18n, useTranslations

packages/core/locales/
  en.json             # engine strings (default)
  zh-CN.json
  es.json
```

- **Server:** `app/layout.tsx` reads locale, wraps children in `I18nProvider`, sets `<html lang>`.
- **Client:** `useTranslations("inbox")` → `t("title")` resolves to `inbox.title`.
- **API:** `getRequestTranslator()` in route handlers; `translateApiError()` maps stable service error codes to `api.*` keys.
- **Engine:** `loadLatestInbox(profile, locale)` re-scores stored runs with the active locale.

## Adding UI strings

1. Add the key to `zh-CN.ts` and `en.ts` (keep shapes identical).
2. Add Spanish in `es.ts` when translating primary surfaces.
3. Use `t("section.key")` or `t("section.key", { name: value })` with `{name}` placeholders in the template.
4. For list joining, use `listSeparator(locale)` — `、` for zh-CN, `, ` for en/es.

## Adding engine strings

1. Add the template to all JSON files under `packages/core/locales/`.
2. Call `translator.t("matcher....", { params })` from matcher/discovery code.
3. Document new keys in [engine-locales.md](./engine-locales.md).

## Locale switcher

`LocaleSwitcher` in the app header toggles **English / 简体中文 / Español**, sets the cookie, and calls `router.refresh()` so server components, API error messages, and match explanations pick up the new locale.
